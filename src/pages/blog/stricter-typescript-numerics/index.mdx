Most TypeScript developers only have to think about one numeric type: `number`. A few might have used `BigInt` as well. But if you work enough with `TypedArray` and `DataView`, you'll know that not all `number` types are equal, which can lead to subtle runtime bugs.

In this post I'll first explain the problem, then walk through how I patch the built-in TypeScript types to improve type safety in my projects.

> **TL;DR:** Run `npx typescript-numeric-type-patch` in your project to get stricter numeric types for `DataView` and `TypedArray`. 

## When `number` is not enough

The problem arises due to the fact that numbers can have different binary representations, while TypeScript just treats them all as the same generic `number` type. Different binary representations have different characteristics, especially related to what range of numbers they can store.

By default, most JavaScript numbers can be assumed to be 64-bit floats. But when working with types such as `DataView` or the various `TypedArray` sub-classes (`Float64Array`, `Int32Array`, `Uint8Array`, etc.), you'll encounter other types of numbers.

Take a look at this code using `TypedArray` for an example:

```typescript
const uint8Array = new Uint8Array([ 123 ]);
// A Uint8 has 8 bits and is unsigned (only positive value),
// so it can store whole numbers between 0 and 255.

const int32Array = new Int32Array([ 123456 ]);
// An Int32 has 32 bits and is signed (both positive and negative values),
// so it can store whole numbers between -2,147,483,648 and 2,147,483,647.

const uint8Value: number = uint8Array[0];
const int32Value: number = int32Array[0];
// As far as TypeScript is concerned, they're the same type.

uint8Array[0] = int32Value;
// Notice! Not enough bits in a Uint8 to store 123456
// It will have to be truncated to some smaller number.

console.log("Value:", uint8Array[0]); 
// Value: 64
// The original value (123456) will repeatedly wrap past the
// max value (255) untill it fits (123456 % 255 = 64).
```

Or an similar example with `DataView`:

```typescript
const buffer = new ArrayBuffer(1000);
const view = new DataView(buffer);
// Create a buffer capable of storing 1000 abitrary bytes of binary data,
// as well as a view into the buffer for easier access.

view.setUint8(0, view.getInt32(1));
// Write the Int32 from offset 1 into offset 0 as a Uint8.
// Assuming that the buffer actually had some values set,
// this would cause the same problem as the previous example.
```

Sometimes this behaviour is indeed what you want. But for most of the use cases I have run into, this is a bug. It can be especially hard to spot once the read and write operations are separated by a few layers of functions.

### Who actually has these issues?

It's a fair question. Extensive use of `TypedArray` and `DataView` is somewhat niche, especially for more mundane web development.

I have personally run into these issues when working on my TypeScript game engine and WebGPU renderer. But you can run into this any time you care about the low-level representation of numbers, which is especially common when performance and memory management are concerns.

## Better declaration for `TypedArray` and `DataView`

To see how I solved this issue, let's first look at the for the type declaration of `DataView`[^data-view]:

[^data-view]: [GitHub - TypeScript - es5.d.ts:1728](https://github.com/microsoft/TypeScript/blob/10b2cf52c310c48e8e632fc0c06f1f0084bf4a0c/src/lib/es5.d.ts#L1728)

```typescript
// Most of the declaration has been omitted for brevity.
interface DataView {
  getInt32(byteOffset: number): number;
  getUint8(byteOffset: number): number;

  setInt32(byteOffset: number, value: number): void;
  setUint8(byteOffset: number, value: number): void;
}
```

If the getters could somehow return more specific types than `number`, and the setters could somehow accept more spefific types as well, then the problem would be solved.

Something like this, perhaps:

```typescript
// Most of the declaration has been omitted for brevity.
interface DataView {
  getInt32(byteOffset: number): Int32;
  getUint8(byteOffset: number): Uint8;

  setInt32(byteOffset: number, value: Int32): void;
  setUint8(byteOffset: number, value: Uint8): void;
}
```

Same goes for `TypedArray` sub-classes such as `Int32Array`[^int32-array], where instead of working with plain `numbers`:

[^int32-array]: [GitHub - TypeScript - es5.d.ts:3273](https://github.com/microsoft/TypeScript/blob/10b2cf52c310c48e8e632fc0c06f1f0084bf4a0c/src/lib/es5.d.ts#L3273)

```typescript
// Most of the declaration has been omitted for brevity.
interface Int32Array {
    at(index: number): number | undefined;
    [index: number]: number;
}
```

More specific types could be used:

```typescript
// Most of the declaration has been omitted for brevity.
interface Int32Array {
    at(index: number): Int32 | undefined;
    [index: number]: Int32;
}
```

### Defining the strict numeric types

At this point you might be wondering about the `Int32` and `Uint8` types I used above. These are not built into TypeScript, so let's dive into how they're defined.

To differentiate between different sub-types of `number`, a technique known as "type branding" is used. The basic isea is to add a fake property to the type being branded, that only exists on the type level.

```typescript
type Uint8 = number & { brand: "Uint8" };

const badUint8: Uint8 = 123;
// Type 'number' is not assignable to type 'Uint8'.
//   Type 'number' is not assignable to type '{ brand: "Uint8"; }'.

const goodUint8: Uint8 = 123 as Uint8;

const genericNumber: number = goodUint8;
// A Uint8 is still a number, it's just a more specific kind of number.
```

With the cast `123 as Uint8`, we're essentially telling TypeScript that the primitive value `123` totally has the property `brand` with a value of `"Uint8"`, even though that is of course not the case at runtime.

We can also see how it helps us distingush between types of numbers:

```typescript
type Uint8 = number & { brand: "Uint8" };
type Int32 = number & { brand: "Int32" };

const uint8: Uint8 = 123 as Uint8;

const int32: Int32 = uint8;
// Type 'Uint8' is not assignable to type 'Int32'.
//   Type 'Uint8' is not assignable to type '{ brand: "Int32"; }'.
//     Types of property 'brand' are incompatible.
//       Type '"Uint8"' is not assignable to type '"Int32"'.
```

The examples above are "strict", in the sense that while a `Int32` _can_ be assigned to a generic `number`, a generic `number` _cannot_ be assigned to an `Int32`.

Having to cast the number literals can be a bit verbose, so you might prefer a more "loose" type of branding:

```typescript
type Uint8 = number & { brand?: "Uint8" };
type Int32 = number & { brand?: "Int32" };
// Notice how the property has now been made optional.

const uint8: Uint8 = 123;
// No longer need a cast.

const int32: Int32 = uint8;
// Type 'Uint8' is not assignable to type 'Int32'.
//   Type 'Uint8' is not assignable to type '{ brand?: "Int32" | undefined; }'.
//     Types of property 'brand' are incompatible.
//       Type '"Uint8" | undefined' is not assignable to type '"Int32" | undefined'.
//         Type '"Uint8"' is not assignable to type '"Int32"'.
```

So while this does allow for bugs where a `number` is accidentally assigned to a `Int32`, it still prevents assigning `Uint8` to `Int32`, which I'm personally much more interested in.

In the rest of this post, I'll use the following utility types for type branding:

```typescript
type StrictlyBranded<T, Name extends string> = T & StrictBrand<Name>;
type LooselyBranded<T, Name extends string> = T & LooseBrand<Name>;

type StrictBrand<Name extends string> = { [__brand]: Name };
type LooseBrand<Name extends string> = { [__brand]?: Name };

declare const __brand: unique symbol;
// Prevent conflicts of the brand property name, such as if you tried
// to brand an object type that already had a "brand" property.

// Usage example:
type Uint8 = LooselyBranded<number, "Uint8">;
type Int32 = LooselyBranded<number, "Int32">;
```

### Why a declaration file won't work

Naively, you might attempt to override the declarations of `DataView` and the `TypedArray` sub-classes using declaration merging in some global `.d.ts` file:

```typescript
type Uint8 = StrictlyBranded<number, "Uint8">;
type Int32 = StrictlyBranded<number, "Int32">;

// Much of the declaration has once again been omitted for brevity
declare global {
  interface Uint8Array {
    at(index: number): Uint8 | undefined;
    [index: number]: Uint8;
  }

  interface Int32Array {
    at(index: number): Int32 | undefined;
    [index: number]: Int32;
  }

  interface DataView {
    getInt32(byteOffset: number): Int32;
    getUint8(byteOffset: number): Uint8;

    setInt32(byteOffset: number, value: Int32): void;
    setUint8(byteOffset: number, value: Uint8): void;
  }
}
```

But this approach unfortunately has a couple of problems. When merging methods of an interface, TypeScript only supports creating overloads[^merging-interfaces]. So while the the new strict method signatures will indeed exist, the basic ones with `number` still work, taking precedence and making it so no type error occurs.

[^merging-interfaces]: [TypeScript Documentation - Declaration Merging - Merging Interfaces](https://www.typescriptlang.org/docs/handbook/declaration-merging.html#merging-interfaces)

<p className="de-emphasize">(If you know of a way to do this with declaration merging or similar, please let me know on <a href="https://bsky.app/profile/tobloef.com">Bluesky</a> or <a href="mailto:tobloef@gmail.com">e-mail</a>. The approach we're about to take in the next section is a lot more hacky than I would like!)</p>

An alternative would be to define wrapped versions of `DataView` and the `TypedArray` sub-classes, and only use those in your code.

```typescript
class TypedDataView {
  private view: DataView;

  // Reimplment the DataView constructor
  constructor(
    buffer: ArrayBufferLike,
    byteOffset?: number,
    byteLength?: number,
  ) {
    this.view = new DataView(buffer, byteOffset, byteLength);
  }

  // Re-implement all the (many) methods of DataView
  getInt32(byteOffset: number, littleEndian?: boolean): Int32 {
    return this.view.getInt32(byteOffset, littleEndian);
  }

  // Etc.
}
```

I dislike this, because it's easy to forget and makes the code less portable. If I want to carry a utility function that uses `DataView` over into another project, that project suddenly has to care about my custom wrapper.

## Patching the built-in types

Because of the reasons outlined above, it appears that there is no way to make these overrides in application-space. Well, if you can't work smarter, work harder.

The approach I have ended up with is to modify the existing declarations files inside the `typescript` package source code. Because I don't want to maintain a fork of TypeScript, I do this by generating a patch file[^patch-files] for the package and applying it after installing TypeScript in a project.

### The patcher script

Modifying the TypeScript declarations by hand is a pain, as you'll have to change hundreds of lines of code in multiple different files.

Therefore, I've written a script[^script] that does this automatically. It essentially just performs crude RegExp replacements in the relevant declaration files, changing out `number` for a more specific type where approriate.

[^script]: [Github - typescript-numeric-type-patch](https://github.com/tobloef/typescript-numeric-type-patch)

Apart from replacing certain uses of `number`, it also defines the various numeric types we need, such as `Int32` and `Uint8`. These will then be globally accessible from any TypeScript file, without having to import them.

### Using the script

The simplest way to run the script is with with `npx` (part of NPM):

```shell frame="none"
# cd ./path/to/your/project
npx typescript-numeric-type-patch
````

The current working directory must be the root folder of your project (the one containing the `node_modules` folder). You can optionally specify the path of your project manually:

```shell frame="none"
npx typescript-numeric-type-patch ./path/to/project
```

If you want "strictly" branded types (where `number` is not assignable to `Int32`), you can add the `--strict` flag.


```shell frame="none"
npx typescript-numeric-type-patch --strict
```

Once the script has run, the declaration files in your `node_modules/typescript` folder will have been modified. 

Depending on your editor, you may have to specify which TypeScript package to use for in-editor hints. In VS Code, you can do this with the `TypeScript: Select TypeScript Version...` command, which you can find in the command palette.

### Generating and applying patch files

Instead of having to run the patch script manually every time you install the dependencies of your project, you can use something like [`patch-package`](https://www.npmjs.com/package/patch-package) if you use Node.js or [`bun patch`](https://bun.com/docs/install/patch) if you use Bun.

With `patch-package`, the flow looks something like this:

```shell frame="none"
npx typescript-numeric-type-patch
npx patch-package typescript
# Remember to add `patch-package` as a `postinstall` script in `package.json`
```

And in Bun, it's something like this:

```shell frame="none"
bun patch typescript
npx typescript-numeric-type-patch
bun patch --commit 'node_modules/typescript'
# No need to add anything to `package.json`, Bun handles that
```

## Results and caveats

After applying the patch, we now have strict numeric types whenever we work with `DataView` and `TypedArray`.

```typescript
const uint8Array = new Uint8Array([ 123 ]);
const int32Array = new Int32Array([ 123456 ]);

uint8Array[0] = int32Array[0];
// Type 'Int32 | undefined' is not assignable to type 'Uint8'.
//   Type 'undefined' is not assignable to type 'Uint8'.
//     Type 'undefined' is not assignable to type 'number'.

const buffer = new ArrayBuffer(1000);
const view = new DataView(buffer);

view.setUint8(0, view.getInt32(1));
// Argument of type 'Int32' is not assignable to parameter of type 'Uint8'.
//   Type 'Int32' is not assignable to type 'StrictBrand<"Uint8">'.
//     Types of property '[__brand]' are incompatible.
//       Type '"Int32"' is not assignable to type '"Uint8"'.
```
